# KIP-185: Make Exactly-Once, In-Order Delivery per Partition the Default Producer Setting — Detailed Summary

**Source:** Apache Kafka wiki (cwiki.apache.org), authored by **Apurva Mehta** (2017, proposed for Kafka 1.0). **Important status nuance:** KIP-185 itself remained *Under Discussion* and was never adopted as written — the community deferred it. The defaults it argued for actually landed years later via **KIP-679 (Kafka 3.0, 2021)**: `enable.idempotence=true` and `acks=all` became the producer defaults. So read KIP-185 as **the argument that defined modern Kafka's defaults**, with KIP-679 as the eventual execution — exactly the "context on modern defaults" it's assigned for.

---

## 1. Motivation: the strongest guarantee shouldn't be opt-in

Pre-change, the Kafka producer's **default** configuration (`acks=1`, `retries=0`, no idempotence) delivered **at-most-once without ordering guarantees** — Kafka's *weakest* semantics — even though, since 0.11 (KIP-98), Kafka's *strongest* guarantee was exactly-once in-order per partition. The KIP's position: with the 1.0.0 milestone approaching, Kafka should ship with its strongest delivery guarantees **out of the box**, so users get safety without having to know which four knobs to turn.

This is the operational-philosophy sequel to "Hands-Free Replication": there, express configs in terms users know; here, **make the safe behavior the default behavior** — users shouldn't need expertise to avoid losing acknowledged data.

## 2. Background definitions (the KIP's precise vocabulary)

**Delivery semantics** (defined over *acknowledged* messages — a sharper framing than usual):

- **At-most-once:** acknowledged messages appear in the log at most once — some acked messages **may be lost**.
- **At-least-once:** every acked message appears at least once — **duplicates possible** from internal retries.
- **Exactly-once:** every acked message appears **exactly once**, no duplicates.

**Failure taxonomy** (the honest fine print):

- **Transient failures:** host recovers fully after the cause passes — power failures, network interrupts, buggy code.
- **Hard failures:** unrecoverable — disk failure, corruption.

The guarantees apply **only to transient failures**; hard failures (or **concurrent transient failures**) can violate even exactly-once.

**Replication interplay (the most quotable technical point):** with **replication-factor = 1, the best possible guarantee is at-most-once regardless of producer settings** — because Kafka flushes to disk *asynchronously*: a power cut or SIGKILL after the ack but before the flush loses acknowledged data. Same logic scales up: with RF=3, a *simultaneous* power outage of all three replicas post-ack, pre-flush loses data. **Durability = replication across failure domains, never fsync** — the design-doc principle restated as a boundary condition on delivery semantics.

## 3. The proposed default changes (the heart of the KIP)

| Config | Old default | Proposed default | Why |
|---|---|---|---|
| `enable.idempotence` | false | **true** | Retries can no longer duplicate (KIP-98 sequence numbers); measured overhead ≈ none |
| `acks` | 1 | **all** | `acks=1` is at-most-once: the leader can ack then crash before replication → acked message lost. Exactly-once *requires* acks=all + idempotence |
| `retries` | 0 | **MAX_INT** | 0 existed only to avoid retry duplicates; idempotence removes the correctness penalty, so retry forever |
| `max.in.flight.requests.per.connection` | 5 | **2** | Pipelining boosts throughput but risked reordering on failure; with KAFKA-5494, sequence numbers preserve order with in-flight > 1. Benchmarks: big gain from 1→2, negligible beyond — so 2 |

**The combined guarantee, stated precisely:** exactly-once, in-order delivery per partition **for topics with RF ≥ 2**, assuming no multiple hard failures or concurrent transient failures.

## 4. Supporting engineering (what had to be true first)

- **KAFKA-5494:** originally the idempotent producer required `max.in.flight=1` for ordering. Broker/client changes let sequence numbers enforce order with **up to 5 in-flight requests** — the fix that made "fast AND ordered" possible, and the reason the modern default keeps in-flight at 5 (≤5 is the safe bound with idempotence).
- **Spurious `OutOfOrderSequenceException` fixes** and **old message-format handling** — with idempotence on by default, the error had to mean what it says. Post-change, receiving `OutOfOrderSequenceException` indicates **previously acknowledged data was actually lost** (e.g., unclean election) — Kafka turning silent data loss into a *reported error*, a detection capability that didn't exist before.
- **Performance analysis:** idempotence alone ≈ free at equal acks/in-flight settings; the real cost is **acks=1 → acks=all latency** (the produce blocks on follower fetches). The KIP's stance: *strong durability out of the box is worth the latency* — a values statement, benchmarked.

## 5. Compatibility notes (why it waited until 3.0)

No migration needed — just default changes — but upgraders would see: (a) the new `OutOfOrderSequenceException`, (b) a changed performance profile from acks=all. That behavioral surprise is largely why the change slipped from 1.0 to KIP-679/Kafka 3.0, which also had to handle a permissions wrinkle (idempotence needs `IDEMPOTENT_WRITE` ACLs; KIP-679 relaxed this so the default wouldn't break locked-down clusters — and clients silently disable idempotence rather than fail when talking to old brokers that can't support it).

**Modern defaults scorecard (Kafka ≥ 3.0):** `enable.idempotence=true` ✅, `acks=all` ✅, `retries=MAX_INT` (with `delivery.timeout.ms=120s` as the real bound, via KIP-91) ✅, `max.in.flight` stayed **5** (not 2 — safe because of KAFKA-5494). So today's producer is **exactly-once in-order per partition by default**; transactions remain the opt-in layer above.

## 6. Important concepts (highlight reel)

- **Defaults are product decisions:** the gap between "strongest guarantee available" and "guarantee users actually get" is a design failure; safe-by-default beats documented-but-optional.
- **Semantics defined over acknowledged messages** — the crisp way to state delivery guarantees (and the definition set later KIPs reuse verbatim).
- **Transient vs hard failures** — every distributed guarantee needs its failure model stated; Kafka's EOS explicitly excludes hard and concurrent failures.
- **RF=1 ⇒ at-most-once, full stop** — async flush means acks without replication are promises without backing; the single most useful line for reasoning about durability configs.
- **`acks=1` is at-most-once** — worth internalizing, since many teams believe acks=1 is "mostly fine": the leader-acks-then-dies window is real data loss of *acknowledged* messages.
- **retries=0 was a duplicates workaround, not a feature** — once idempotence exists, infinite retries are strictly better (delivery.timeout.ms bounds them in time instead).
- **Pipelining vs ordering resolved by sequence numbers** — the in-flight story (1 → proposed 2 → actual 5) shows how a correctness mechanism (KIP-98 sequences) can *buy back* a performance feature.
- **`OutOfOrderSequenceException` as a data-loss detector** — the exception isn't noise; it's Kafka telling you acknowledged data vanished.
- **KIP-185 → KIP-679 lineage** — proposals can "lose" and still win: the argument was accepted, the timing wasn't.

## 7. Real-world use cases — where this matters

**Directly, as the engineer configuring producers:**

- **Knowing what you get for free (Kafka ≥ 3.0):** any Spring Boot/Java service producing events today already has exactly-once-per-partition — no dedup keys, no manual `enable.idempotence` — *provided* RF ≥ 2 (ideally 3) and `min.insync.replicas=2`. For an order/payment event producer (an OrderFlow- or PaySwitch-style service), the KIP explains why the modern happy path is correct without extra code.
- **Auditing legacy configs:** clusters/apps born before Kafka 3.0 often carry explicit `acks=1`/`retries=3` from old tutorials — this KIP is the checklist for why those overrides should be deleted, and what changes when you do (latency up slightly, silent loss gone).
- **Incident interpretation:** an `OutOfOrderSequenceException` in production = investigate for real data loss (unclean election? RF=1 topic?), not just retry-and-ignore.
- **Capacity/latency planning:** the acks=1→all latency delta is the known, accepted cost; if a metrics/telemetry topic genuinely prefers speed over durability, *explicitly* opting down (acks=1 or 0) is now a documented deviation rather than an accidental default.
- **Interview framing:** "What guarantees does a default Kafka producer give, and since when?" — the complete answer cites KIP-98 (mechanism), KIP-185 (argument), KAFKA-5494 (ordering with pipelining), KIP-679 (shipped in 3.0), and the RF≥2 + no-concurrent-failures fine print. That layered answer is a senior-level differentiator.

**As a transferable design lesson:**

- **Safe-by-default API design:** any client library or service you build should default to the strongest correctness mode whose cost is acceptable, and make *weakening* explicit (your Redis caching layer, HTTP client retry policies, DB drivers). KIP-185/679 is the reference story for arguing that in a design review.
- **Stating the failure model with the guarantee:** "exactly-once (transient failures, RF≥2, no concurrent outages)" is the template for honestly documenting any guarantee your own system offers.

## 8. TL;DR

KIP-185 (2017) argued that Kafka's producer should default to its *strongest* guarantee — exactly-once, in-order per partition — instead of its weakest (at-most-once, unordered): flip `enable.idempotence` to true (free, thanks to KIP-98 sequence numbers), `acks` to all (acks=1 loses acknowledged data when a leader dies pre-replication), `retries` to MAX_INT (idempotence removed the duplicate penalty), and cap in-flight requests where sequence numbers preserve order (KAFKA-5494). It also stated the fine print with rare precision: guarantees cover *acknowledged* messages under *transient* failures only, and with replication-factor 1, async flushing makes at-most-once the ceiling no matter what the producer does. The KIP itself stalled, but its argument won: **KIP-679 shipped these defaults in Kafka 3.0 (2021)** — idempotence on, acks=all, in-flight kept at 5 — so every modern Kafka producer is exactly-once per partition out of the box, and `OutOfOrderSequenceException` now means "acknowledged data was actually lost." The meta-lesson pairs with the Hands-Free Replication post: first express configs in terms users know; then make the safe configuration the one users get without asking.
